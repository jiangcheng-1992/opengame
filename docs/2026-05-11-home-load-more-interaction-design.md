# 首页查看更多交互设计

## 问题
- 当前首页和“我的作品”底部使用“下一页”链接，点击后会整页跳转。
- 对游戏广场这种浏览流来说，分页会打断用户正在看的上下文，移动端尤其明显。
- 这个问题本质不是分页数据能力不足，而是列表浏览心智与“下一页”网页导航不匹配。

## 目标
- 将底部“下一页”改为“查看更多作品”，点击后在当前列表下方追加下一批作品。
- 保留首屏服务端渲染，保证首页首次打开速度和不可用兜底逻辑不变。
- 复用现有 `/api/games` 与 cursor 分页能力，不改数据库、生成链路、作品状态机。

## 推荐方案
- 新增一个客户端作品流组件，接收首屏 `games`、`nextCursor`、当前 `tab` 和 `status`。
- 首屏内容仍由 `app/page.tsx` 调用 `listGames` 获得，再交给客户端组件渲染。
- 点击“查看更多作品”时请求 `/api/games?tab=...&status=...&cursor=...`，将返回的作品追加到已有列表。
- 加载中按钮文案为“加载中...”，避免重复点击。
- 请求失败时保留当前列表，在底部显示简短失败原因和“重试”按钮。
- `nextCursor` 为空时隐藏按钮，或显示轻量的“已经到底了”状态。

## 非目标
- 不做自动无限滚动，避免用户失去页尾感，也降低加载失败和位置恢复复杂度。
- 不新增页码、上一页、排序、筛选或搜索。
- 不改变内置精选与公开作品的混排规则。
- 不引入假作品、假加载结果或本地缓存伪造状态。

## 用户影响
- 用户滑到底后可以继续看下一批作品，不会跳回页面顶部。
- 当前浏览上下文保留，尤其适合移动端连续浏览。
- 加载失败时用户仍留在原列表，可以原地重试。

## 实现范围
- `app/page.tsx`：保留数据获取和空状态判断，把有数据时的列表交给客户端作品流组件。
- `components/`：新增作品流组件，管理追加加载、错误、重试和按钮状态。
- `app/globals.css`：调整底部加载区域样式，确保桌面和移动端按钮不拥挤、不溢出。

## 验证
- 基础验证：`npm run lint`、`npm run build`、`npx prisma generate`。
- UI 验证：本地打开首页和“我的作品”，检查桌面与移动端列表追加、加载中、失败重试、无更多作品状态。
- 回归重点：空状态、数据库不可用状态、内置精选展示、`tab=mine` 与状态筛选不受影响。
