# 作者工作态与公共游玩态分流设计

## 背景

同一个游戏详情页同时承载公共试玩、作者继续修改、生成进度和对话历史，导致主态和客态混在一起。用户从首页点击游戏时，任务是试玩；用户从自己的创作台点击作品时，任务是查看创作状态或继续修改。

本次调整的核心原则是：入口代表用户任务，页面只服务当前任务。

## 决策

| 来源 | 状态 | 点击目标 | 页面任务 |
|---|---|---|---|
| 公共 Gallery | `READY` / 内置精选 | `/games/:id` | 纯试玩 |
| 我的作品 / 创作台 | `DRAFT` | `/create?game=:id` | 继续一创头脑风暴 |
| 我的作品 / 创作台 | `GENERATING` | `/create?game=:id` | 查看一创生成进度 |
| 我的作品 / 创作台 | `READY` | `/games/:id/edit` | 试玩当前版本并继续修改 |
| 我的作品 / 创作台 | `FAILED` | `/games/:id/edit` | 查看失败原因，调整方向后重新生成 |

一创生成成功后也进入 `/games/:id/edit`，因为此时用户仍处在作者工作态，下一步自然是试玩自己的版本并继续打磨。

## 页面边界

### 公共游玩页 `/games/:id`

- 只服务试玩。
- 首屏保留游戏 iframe、标题、状态、播放数和 Like。
- 保留玩法说明、摘要、标签和内置精选说明。
- 不展示继续修改入口、生成进度日志、创作对话历史或作者工作流解释。
- 公共 Gallery 只展示 `READY` 作品和内置精选；未完成或失败作品不进入公共试玩流。

### 一创工作台 `/create?game=:id`

- 承接 `DRAFT` 和 `GENERATING`。
- `DRAFT` 继续头脑风暴和 brief 收束。
- `GENERATING` 原地展示生成、自动试玩、自动修复、发布进度。
- 失败后不再停留在公共详情页，创作台卡片会把 `FAILED` 作品送入修改工作台。

### 修改 / 修复工作台 `/games/:id/edit`

- 承接作者自己的 `READY` 和 `FAILED` 作品。
- `READY`：左侧试玩当前版本，右侧整理修改计划并生成新版本。
- `FAILED`：左侧显示最近失败原因和日志，右侧允许用户写修复方向并重新生成。
- 对 `FAILED` 重新生成时，不要求已有 `sourceUrl`；后端使用原始对话、确认 brief、最近失败原因和用户修复指令重新构造生成 prompt。

## 技术范围

- `GameCard` 增加来源上下文，公共 Gallery 和创作台使用不同跳转规则。
- `listGames("all")` 只返回公开且 `READY` 的真实作品，继续合并内置精选。
- `CreateGameForm` 支持从服务端恢复正在生成的草稿 Job，并在完成后进入修改工作台。
- `/api/games/:id/messages` 允许作者对 `FAILED` 且无可玩版本的作品发起修复式重新生成。
- `/games/:id` 移除作者操作、生成日志和对话历史展示。
- `/games/:id/edit` 增加失败修复态。

## 不做

- 不新增数据库表、版本历史表或独立作者详情页。
- 不恢复 Remix / 二创公共入口。
- 不把公共详情页变成自动跳转页；直达 `/games/:id` 始终是纯游玩语义。
- 不改变 OpenGame、GitHub Actions、Sandbox、自动试玩验证链路。

## 验证

- `npx prisma generate`
- `npm run lint`
- `npm run build`
- 本地打开 `/`，确认公共 Gallery 卡片进入纯游玩页，页面不展示生成日志和创作对话历史。
- 本地打开 `/?tab=mine`，确认不同状态作品进入对应工作台。
- 本地打开 `/games/:id/edit`，确认 `READY` 可继续修改，`FAILED` 可看到失败原因并重新生成。
