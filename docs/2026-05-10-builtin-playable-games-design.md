# 内置可玩游戏设计

## 背景

新用户首次进入作品广场时，如果本地数据库、Vercel Blob、OpenGame 或 MiniMax 凭据不可用，页面会呈现空状态。空状态符合“不用 mock 冒充完成”的规则，但无法让用户立即理解产品的核心价值：点开一个 HTML5 游戏试玩，再去创建自己的游戏。

本设计引入 21 款已完成封面和试玩入口的静态内置游戏，作为 onboarding playable catalog。它们是项目内置内容，不写入 Prisma，不伪装成 OpenGame 真实生成结果。

## 方案

- `lib/builtin-games.ts` 维护内置游戏清单、元数据、试玩路径、封面路径和玩法 brief。
- `public/builtin-games/<slug>/index.html` 放每款可玩的 HTML 入口。
- `public/builtin-games/shared/engine.js` 放共享 Canvas 小游戏运行时。
- `public/builtin-games/<slug>/cover.png` 放通过生图技能生成的位图封面，不使用 SVG 封面。
- 作品广场的 `all` tab 混入内置游戏；数据库不可用时仍显示内置游戏。
- `mine` tab 仍只显示当前匿名用户真实作品和草稿。
- 详情页支持内置游戏播放，并显示“内置精选”来源标识。
- 内置游戏只承担 onboarding 试玩，不提供 Remix 生成入口。

## 用户影响

- 首屏不再空白，用户可以直接点开试玩 21 款不同玩法。
- 用户能明确区分“内置精选”和“真实生成作品”，避免误解平台已经生成了这些作品。
- 没有数据库或第三方凭据时，试玩体验仍可用；创建 / 继续修改才依赖真实后端链路。

## 错误处理

- 数据库不可用时，作品广场展示内置精选，不阻塞试玩。
- 内置游戏 Like 使用本地前端状态，不写数据库。
- 内置游戏 playCount 使用静态基线展示，不写数据库。
- 内置游戏不触发生成任务，不使用假任务冒充成功。

## 验证

- 本地基础验证：`npm run lint`、`npm run build`、`npx prisma generate`。
- UI 验证：启动本地服务，打开首页确认 21 款内置游戏可见；打开至少一款详情页确认 iframe 可加载、点击/键盘输入能改变游戏状态。
