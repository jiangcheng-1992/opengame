# 统一创作与修改工作台设计

## 1. 决策

游戏第一次创作和第二次修改统一为同一套创作语法，但不强行做成相同布局。

- 新游戏创作是 Idea-first Workbench：用户先表达想法，再通过对话澄清需求。
- 修改游戏是 Playtest-first Workbench：用户先试玩当前版本，发现问题后在右侧整理修改计划。
- 修改场景默认心智是“改这个游戏”，不是 Remix，也不是重新做一个游戏。
- 二改左侧只负责试玩，不做问题标记、截图圈选或测试反馈系统。
- 二改右侧采用计划优先：保留项、变更项、待确认常驻在对话上方。

## 2. 信息架构

### 新游戏创作

- 左侧主区：流式对话、可点回复、输入框。
- 右侧面板：需求进度、最终 brief、公开/私密、生成可玩版本。
- 四个需求槽位保持为核心玩法、操作方式、胜负目标、视觉风格。

### 修改游戏

- 顶部：返回详情、游戏名、当前状态、旧版本保留提示。
- 左侧主区：当前版本 iframe 试玩；下方只放轻量版本信息。
- 右侧置顶：修改计划，包括保留项、变更项、待确认。
- 右侧中部：修改对话和建议回复。
- 右侧底部：修改 brief、生成新版本。

## 3. 交互规则

- 详情页不再承载完整二改输入框，只提供进入修改工作台的入口。
- 二改提交前必须形成可读的修改 brief，明确“保留什么、改变什么”。
- 二改最多追问必要信息，不把用户拖回完整一创四槽位流程。
- 生成新版本时继续复用现有 `/api/games/:id/messages` 生成链路，并保留旧版本可玩。

## 4. 不做

- 不恢复 Remix 入口。
- 不新增数据库表或版本历史 UI。
- 不新增截图标注、问题分类按钮、画面圈选或时间点记录。
- 不改同源 iframe 播放代理和真实生成链路。

## 5. 验证

- `npx prisma generate`
- `npm run lint`
- `npm run build`
- 本地打开 `/create` 和 `/games/:id/edit` 检查桌面与移动布局：
  - 一创仍以对话和需求进度为核心。
  - 二改首屏左侧可试玩，右侧计划优先。
  - 详情页继续修改入口能跳转到二改工作台。
  - 页面无横向滚动，主要按钮和输入框在移动端可操作。
