# Bright Arcade UI/UX Redesign

> 2026-05-10 更新：二创 / Remix 已从当前 MVP 范围移除，详见 `docs/2026-05-10-remove-remix-scope-design.md`。本文中 Remix 相关入口和配色只作为早期探索记录，不再作为实现依据。

## 1. 目标

本轮重做把产品从“生成器后台页面”改成“明亮、低门槛、可试玩的游戏厅”。第一屏优先回答三个问题：

1. 这里有什么可以马上玩？
2. 我可以怎样快速创建一个自己的版本？
3. 看到喜欢的作品后如何 Remix 或继续修改？

视觉方向采用明亮玩具感：浅色底、清晰卡片、强行动按钮、多色点缀。它服务的是低门槛和可扫读，不做纯营销 Hero，也不做儿童化装饰。

## 2. 信息结构

首页以 Arcade 试玩优先：

- Featured game：优先展示可玩的高信号作品，封面和 Play 入口最大。
- Quick create：保留 prompt 入口，但作为“从灵感开始”的行动面板，不压过作品。
- Game shelves：按 Players' Choice、Fresh Builds、Remix Ready 分组，用真实作品信号排序。
- Empty/unavailable state：不使用假作品；数据库不可用或没有作品时，只展示真实状态和创建入口。

创建页改为单屏工作台：

- Prompt 是主输入，不再被大段说明文字推到屏幕下方。
- 示例 prompt、可见性、生成按钮都围绕同一条创建路径。
- 文案只保留能直接改善生成质量的提示。

详情页采用 stage-first：

- 游戏 iframe 是主角，标题和元信息压缩到舞台栏。
- Remix、继续修改、Like 作为清晰行动区。
- 生成进度、日志、对话历史降级为辅助面板，避免压过可玩内容。

## 3. 数据包装

`Game` 新增展示元数据：

- `summary`：一句话说明玩法。
- `genre`：用于扫读和分组的类型。
- `tags`：少量玩法标签。
- `controls`：操作方式标签。
- `coverPrompt`：封面生成提示词。

创建和 Remix 请求体不变。服务端在创建作品时尝试用 MiniMax 生成元数据；失败或没有凭据时使用规则兜底。元数据和封面都是增强项，不能阻塞真实 OpenGame 生成任务。

已有作品通过可重复执行的 backfill 脚本补齐元数据。脚本有 MiniMax 凭据时尝试 AI 包装；没有凭据时只写规则兜底结果，不伪造 AI 来源。

## 4. 视觉原则

- 浅色主背景，避免当前黑绿工程感。
- 单个卡片圆角不超过 8px，强调可点击和可播放。
- 使用多色但有节制的语义点缀：绿色用于创建/成功，蓝色用于试玩，红橙用于 Remix/热度。
- 首屏不放大段产品解释；文字只服务操作。
- 移动端底部导航必须避开主按钮和输入框，详情页优先露出游戏画布。

## 5. 验证

基础验证：

- `npx prisma generate`
- `npm run lint`
- `npm run build`

UI 验证：

- 桌面 1280x720：首页、创建页、详情页首屏行动清楚。
- 移动 390x844：无横向滚动、无底部导航遮挡、详情页先看到游戏画布和主要操作。

真实链路：

- 凭据可用时生成一个最小游戏，确认同源 `/api/games/:id/files/index.html` 可 iframe 播放。
- MiniMax 元数据或封面失败时，作品仍能进入生成任务和详情页。
