# 创作路径内联生成进度设计

## 背景

创建和继续修改游戏在确认意图后会跳到详情页，通过详情页的进度日志表达生成状态。这个跳转没有带来新的有效信息，反而打断了用户刚建立的创作上下文。

本次调整的核心原则是：生成进度不是一个新的用户任务，它只是创作任务的等待状态。用户在哪里确认生成，就应该在哪里等待结果。

## 决策

- 创建页和修改页启动生成后不再跳转到详情页等待。
- 创建页在右侧 brief 面板内展示生成状态。
- 修改页在修改 brief 卡片下方展示生成状态，左侧当前版本继续可试玩。
- 已可玩作品提交修改后不把 `Game.status` 改成 `GENERATING`；新版本进度由最新 Job 表达，旧版本继续作为当前可玩版本展示。
- 生成成功后自动跳到 `/games/:id/edit`，进入作者工作台试玩并继续打磨。
- 一创生成失败后进入 `/games/:id/edit` 修复工作台，保留 brief、失败原因和日志。
- 修改生成失败后停留 `/games/:id/edit`，保留修改计划，并显示失败原因和日志。
- 公共详情页不再展示生成进度；生成进度只属于创作页和修改工作台。

## 用户路径

### 创建

```text
首页/创建入口
  -> 创建工作台：对话澄清意图
  -> 用户确认 brief
  -> 原地进入生成中状态
  -> 成功：自动进入修改工作台试玩
  -> 失败：进入修改工作台，显示原因并允许调整后重新生成
```

### 继续修改

```text
修改工作台试玩
  -> 修改工作台
  -> 左侧试玩当前版本，右侧整理修改计划
  -> 用户确认生成新版本
  -> 原地进入新版本生成中状态，当前版本仍可试玩
  -> 成功：自动回到修改工作台试玩新版本
  -> 失败：原地显示失败原因，保留修改计划
```

## 交互细节

- 主状态面向普通用户，只显示排队中、生成中、自动试玩中、自动修复中、发布中、完成、失败。
- 原始日志默认作为辅助信息展示在同一模块里，不作为页面主标题。
- 生成中禁用重复提交按钮，避免同一 brief 重复创建 Job。
- 成功跳转只在 `done` 状态触发；一创失败会跳到修复工作台，修改失败留在当前修改工作台。
- 修改页生成等待时，当前可玩版本不被替换，`Game.status` 保持 `READY`，避免用户失去二创参照物。

## 技术方案

- `JobWatcher` 增加可选跳转目标，任务完成后可自动 `router.push`。
- `CreateGameForm` 保存 `activeJobId`，启动生成后不再 `router.push(/games/:id?job=...)`。
- `EditGameWorkbench` 保存 `activeJobId`，启动生成后不再离开编辑页。
- 后端生成链路、Job 状态机、数据库结构保持不变。

## 不做

- 不新增独立生成页。
- 不新增 Job 表或版本历史表。
- 不改变 OpenGame、GitHub Actions、Sandbox、自动试玩验证链路。

## 验证

- `npx prisma generate`
- `npm run lint`
- `npm run build`
- 本地打开 `/create` 检查确认生成后是否原地出现进度模块。
- 本地打开 `/games/:id/edit` 检查生成新版本后是否保留试玩区并原地显示进度模块，完成后仍回到作者工作台。
