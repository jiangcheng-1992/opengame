# 页面切换性能优化设计

## 背景
- 用户反馈：页面切换时响应慢、卡顿，尤其是首页、作品详情、创建页之间来回切。
- 只读诊断事实：
  - 内置游戏封面 21 张，合计约 49MB，单张约 2.0-2.8MB。
  - 首页卡片使用 CSS `backgroundImage` 展示封面，缺少浏览器原生懒加载和 Next 图片优化。
  - 首页和详情页均为动态服务端渲染；每次切换都要重新执行服务端查询。
  - 详情页一次性读取完整对话历史和最新任务，低优先级内容会跟首屏竞争。

## 目标
- 让首页与详情页切换更轻、更快，优先改善用户可感知等待。
- 不改变 MVP 产品闭环：Gallery、我的作品、创建、详情播放、继续修改、Like、playCount 继续可用。
- 不用 mock 数据冒充完成，真实生成链路规则不变。

## 方案
1. 封面资源优化
   - `GameCard` 使用 `next/image` 渲染封面，保留稳定 `16:9` 容器。
   - 首屏少量卡片设置较高加载优先级，其余卡片懒加载。
   - 批量压缩内置 `cover.png`，保留原文件名和目录约定。

2. 首页数据轻量化
   - 首页列表查询只返回卡片需要的字段，避免把不必要的大字段或完整关系带到首屏。
   - 保留最新任务和当前用户 Like 状态，保证草稿、生成中、失败状态仍能正确展示。

3. 详情页首屏减负
   - 详情页主查询只取播放、标题、统计、权限、最新任务等首屏字段。
   - 对话历史只取最近若干条，避免长对话把首屏拉慢。
   - iframe 仍使用同源代理，不回退到 Blob 直链。

4. 验证
   - 本地执行 `npm run lint`、`npm run build`、`npx prisma generate`。
   - 启动本地服务，打开首页和详情页检查卡片、导航、iframe 播放区域。
   - 验证通过后部署线上，供用户在线上体验。

## 取舍
- 不引入新缓存层或复杂分页机制；当前瓶颈主要是资源体积和首屏查询过重，先走最短路径。
- 不删除原有内置游戏，也不改变作品排序和可见性规则。
- 不把详情页对话历史完全异步拆 API，避免本轮改动扩大接口面。
